**Change 1: Status filter on GET /**

Add query parameter `status` to the GET / endpoint. Default behavior: return only discovered themes (`discoveredAt IS NOT NULL`). Supported values:
- `discovered` (default): `where: { userId, discoveredAt: { not: null } }`
- `pending`: `where: { userId, discoveredAt: null }`
- `all`: `where: { userId }` (no discoveredAt filter)

Parse status from `req.query.status` as string, default to `'discovered'`. Build the Prisma `where` clause accordingly.

**Change 2: Progress aggregation on GET /**

After fetching themes, run a single raw SQL query to compute per-theme progress data. Add this AFTER the existing `findMany`:

```typescript
const progressData = await prisma.$queryRaw<{
  themeId: string;
  totalCards: bigint;
  masteredCards: bigint;
  dueCards: bigint;
}[]>`
  SELECT
    t.id AS "themeId",
    COALESCE(COUNT(card.id), 0) AS "totalCards",
    COALESCE(COUNT(card.id) FILTER (WHERE card.repetitions >= 3), 0) AS "masteredCards",
    COALESCE(COUNT(card.id) FILTER (WHERE card."nextReviewAt" <= NOW()), 0) AS "dueCards"
  FROM "Theme" t
  LEFT JOIN "ContentTheme" ct ON ct."themeId" = t.id
  LEFT JOIN "Quiz" q ON q."contentId" = ct."contentId" AND q."isSynthesis" = false
  LEFT JOIN "Card" card ON card."quizId" = q.id AND card."userId" = ${userId}
  WHERE t."userId" = ${userId} AND t."discoveredAt" IS NOT NULL
  GROUP BY t.id
`;
```

Note: Use `card.repetitions >= 3` as mastery threshold (per research recommendation -- works for early users, doesn't require 21+ day intervals).

Build a `Map<string, { totalCards, masteredCards, dueCards, masteryPercent }>` from progressData. Merge into the response:

```typescript
const progressMap = new Map(progressData.map(p => [
  p.themeId,
  {
    totalCards: Number(p.totalCards),
    masteredCards: Number(p.masteredCards),
    dueCards: Number(p.dueCards),
    masteryPercent: Number(p.totalCards) > 0
      ? Math.round(Number(p.masteredCards) / Number(p.totalCards) * 100)
      : 0,
  },
]));
```

In the result mapping, spread progress data:
```typescript
const progress = progressMap.get(theme.id) || { totalCards: 0, masteredCards: 0, dueCards: 0, masteryPercent: 0 };
return { ...theme, contentCount: _count.contentThemes, tags: ..., ...progress };
```

Only run the progress query when status is `'discovered'` or `'all'` (not for `'pending'` since pending themes have no progress).

**Change 3: POST /discover endpoint**

Add a new Zod schema and endpoint BEFORE the existing POST / route:

```typescript
const discoverActionSchema = z.discriminatedUnion('type', [
  z.object({ type: z.literal('confirm'), themeId: z.string() }),
  z.object({ type: z.literal('rename'), themeId: z.string(), newName: z.string().min(1).max(100).trim() }),
  z.object({ type: z.literal('merge'), sourceThemeId: z.string(), targetThemeId: z.string() }),
  z.object({ type: z.literal('dismiss'), themeId: z.string() }),
]);

const discoverSchema = z.object({
  actions: z.array(discoverActionSchema).min(1).max(50),
});
```

Endpoint logic for `POST /discover`:

1. Parse body with `discoverSchema.parse(req.body)`
2. Open a `prisma.$transaction` and process each action:

   - **confirm**: `tx.theme.update({ where: { id: themeId, userId }, data: { discoveredAt: new Date() } })`
     Verify ownership by including userId in the where clause (use `findFirst` + `update` pattern if needed since `update` requires unique field).

   - **rename**: Find the theme (`findFirst where { id, userId }`), generate new slug via `generateSlug(newName)`, check slug uniqueness for user (excluding current theme), then update `{ name: newName, slug, discoveredAt: new Date() }`.

   - **merge**:
     a. Verify both source and target themes belong to userId
     b. Move ContentTheme records: find all `contentTheme` where `themeId = sourceThemeId`, for each upsert into targetThemeId (skip if already exists via `upsert`)
     c. Move ThemeTag records: find all `themeTag` where `themeId = sourceThemeId`, for each upsert into targetThemeId
     d. Delete synthesis quizzes from source: `tx.quiz.deleteMany({ where: { themeId: sourceThemeId, isSynthesis: true } })`
     e. Delete source theme: `tx.theme.delete({ where: { id: sourceThemeId } })`
     f. Confirm target: `tx.theme.update({ where: { id: targetThemeId }, data: { discoveredAt: new Date() } })`
     g. Clear memo cache on target: `{ memo: null, memoGeneratedAt: null }`

   - **dismiss**: Verify ownership, then `tx.theme.delete({ where: { id: themeId } })` (cascade deletes ContentTheme, ThemeTag, Quiz join records).

3. After transaction, fetch updated themes for user (discovered only) and return `{ themes: [...] }`.

Handle Zod errors with 400 response. Log the action summary.

Important: The POST /discover route MUST be registered before POST / (which creates a new theme) to avoid Express matching `/discover` as an `:id` parameter on POST /:id routes. Since POST / and POST /discover are both POST on different paths, Express should match them correctly, but place `/discover` first to be safe.

Also update the manually-created theme POST / endpoint: when a user manually creates a theme, set `discoveredAt: new Date()` so it's immediately visible (user-created themes skip discovery).
