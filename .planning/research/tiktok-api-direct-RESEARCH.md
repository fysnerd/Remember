# Research: TikTok API Direct Call Approach

**Researched:** 2026-02-10
**Mode:** Feasibility
**Overall confidence:** MEDIUM

## Executive Summary

Replacing TikTok's full DOM scraping with a direct API call is **feasible but requires a different approach than Instagram**. Unlike Instagram (where `context.request.get()` works with simple mobile headers), TikTok's `favorite/item_list` endpoint requires a cryptographic signature (X-Bogus) generated by TikTok's own JavaScript VM running in the browser. You cannot call this endpoint with raw HTTP requests -- the signature validation will reject them. However, you CAN eliminate scrolling entirely by using `page.evaluate()` to execute a `fetch()` call from within the browser context, leveraging TikTok's own patched `window._fetch` and signing infrastructure. This approach would reduce sync time from ~60s to ~10-15s per user while keeping the browser requirement.

**Verdict: YES, with conditions.** Use the `page.evaluate(fetch(...))` approach, not the `context.request` approach. The browser is still needed, but scrolling and tab-clicking are eliminated entirely.

## Current Architecture

### How tiktokSync.ts Works Today

1. **Launch Chromium** (headless, 1920x1080 desktop viewport)
2. **Inject cookies** from stored `TikTokCookies` (sessionid, msToken, tt_csrf_token, etc.)
3. **Navigate** to `https://www.tiktok.com/profile` and wait for `networkidle`
4. **Wait 5 seconds** for dynamic content to load
5. **Close cookie popup** if present
6. **Click "Liked" tab** using 5 different selector strategies (text, French text, role, data-e2e, XPath)
7. **Set up response interception** via `page.on('response')` to capture `favorite/item_list` API responses
8. **Scroll 15 times** with 1.5s delays, collecting intercepted API responses containing `itemList` arrays
9. **Deduplicate and save** new videos to the database
10. **Close browser**

**Time per user:** ~60s (5s page load + 5s wait + 5s tab click + ~22s scrolling + overhead)

**Pain points:**
- 5 methods to click "Liked" tab (fragile, locale-dependent)
- Scrolling is slow and unreliable (may not trigger API calls)
- Large screenshots saved for debugging
- `foundExisting` check happens during scrolling (inefficient)

### How instagramSync.ts Works (the target pattern)

1. **Launch Chromium** (headless, mobile viewport, anti-detection args)
2. **Inject cookies**, navigate to `https://i.instagram.com/`
3. **Validate session** (check if redirected to login)
4. **Call API directly** via `context.request.get('https://i.instagram.com/api/v1/feed/liked/', { headers: { 'User-Agent': 'Barcelona...', 'X-IG-App-ID': '...', 'X-CSRFToken': '...' } })`
5. **Parse JSON response**, filter videos, save to DB
6. **Close browser**

**Time per user:** ~10s

**Why this works for Instagram:** Instagram's mobile API (`i.instagram.com`) does not require client-side cryptographic signatures. It validates: (1) session cookies, (2) the Barcelona mobile User-Agent, (3) the X-IG-App-ID. That's it. Playwright's `context.request` sends the browser's cookies with custom headers, bypassing the need for any signature.

## TikTok API Analysis

### Endpoint Details

**URL:** `https://www.tiktok.com/api/favorite/item_list`

**Query Parameters:**
| Parameter | Value | Description |
|-----------|-------|-------------|
| `secUid` | User's secure UID | Identifies the user whose likes to fetch |
| `count` | 35 | Number of items per page |
| `cursor` | 0 (then from response) | Pagination offset |

**Response Format:**
```json
{
  "itemList": [
    {
      "id": "7123456789",
      "desc": "Video description...",
      "author": { "uniqueId": "username", "nickname": "Display Name" },
      "stats": { "playCount": 1000, "diggCount": 50, "commentCount": 10, "shareCount": 5 },
      "video": { "duration": 30, "cover": "https://..." },
      "createTime": 1700000000
    }
  ],
  "hasMore": true,
  "cursor": "1700000001000"
}
```

**Sources:** [davidteather/TikTok-Api user.py](https://github.com/davidteather/TikTok-Api/blob/main/TikTokApi/api/user.py), intercepted responses in current code. **Confidence: HIGH.**

### Security Mechanisms

TikTok implements multiple layers of anti-bot protection:

| Mechanism | What It Does | Can We Bypass? |
|-----------|-------------|----------------|
| **X-Bogus** | Cryptographic signature on URL + user-agent + timestamp. Generated by `window.byted_acrawler.frontierSign()` in TikTok's JS VM. | Only from within browser context |
| **X-Gnarly** | Secondary header signature combining msToken + User-Agent + signed URL | Only from within browser context |
| **msToken** | Session token in cookies, validated server-side | Already have from user's cookies |
| **Behavioral detection** | Mouse movement, scroll patterns, timing analysis | Irrelevant for API-only approach |
| **TLS fingerprinting** | Server checks TLS client hello characteristics | Playwright Chromium passes this |
| **IP reputation** | Datacenter IPs flagged more aggressively | VPS risk, but current setup works |

**Critical insight:** TikTok's `window._fetch` is a **patched version** of the native `fetch()`. When TikTok's own frontend code calls `fetch()` to load liked videos, TikTok's JavaScript interceptor automatically appends X-Bogus, X-Gnarly, and other required signatures. A `fetch()` call executed via `page.evaluate()` goes through this same patched pipeline.

**Sources:** [int4444/tiktok-api](https://github.com/int4444/tiktok-api), [carcabot/tiktok-signature](https://github.com/carcabot/tiktok-signature/issues/131), [davidteather/TikTok-Api tiktok.py](https://github.com/davidteather/TikTok-Api/blob/main/TikTokApi/tiktok.py). **Confidence: HIGH** for X-Bogus requirement, **MEDIUM** for `_fetch` auto-signing behavior.

### Signature Requirements

The X-Bogus algorithm (documented by [int4444/tiktok-api](https://github.com/int4444/tiktok-api)):

1. **Triple MD5 hashing** of URL parameters, body, and user-agent
2. **RC4 encryption** as preprocessing step
3. **Salt array assembly** with timestamp + magic numbers + hash fragments
4. **Custom Base64 encoding** with shifted alphabet: `"Dkdpgh4ZKsQB80/Mfvw36XI1R25-WUAlEi7NLboqYTOPuzmFjJnryx9HVGcaStCe"`

Reproducing this in Node.js is theoretically possible but impractical:
- TikTok updates the algorithm periodically (breaking reverse-engineered implementations)
- The algorithm runs inside an obfuscated JavaScript VM with >77 opcodes
- Even the most sophisticated open-source implementations (davidteather's TikTok-Api v7.2.1) use **browser-based signing**, not native reimplementation

## Approach Comparison

### A: context.request (Instagram-style)

**How it would work:** After browser session validation, call `context.request.get('https://www.tiktok.com/api/favorite/item_list?secUid=...&count=35&cursor=0')` with custom headers.

| Criterion | Rating |
|-----------|--------|
| **Difficulty** | 1/5 (trivial code change) |
| **Reliability** | 1/5 (will NOT work) |
| **Speed improvement** | N/A |
| **Risk** | Guaranteed failure |

**Why it fails:** Playwright's `context.request` makes raw HTTP requests from Node.js with the browser's cookies. These requests bypass TikTok's JavaScript VM entirely. No X-Bogus signature is generated. TikTok's server will reject the request with a 403 or return empty results.

This is the fundamental difference from Instagram: Instagram's mobile API does not require client-side cryptographic signatures. TikTok's does.

**Verdict: DO NOT USE.**

### B: page.evaluate + fetch (recommended)

**How it would work:** Navigate to TikTok profile, wait for TikTok's JS to load (including `byted_acrawler`), then call the API from within the browser's JavaScript context using `page.evaluate()`.

```typescript
// After navigating to profile and waiting for TikTok's JS to load:
const result = await page.evaluate(async (params) => {
  const url = `https://www.tiktok.com/api/favorite/item_list?secUid=${params.secUid}&count=35&cursor=0`;
  const response = await fetch(url);
  const data = await response.json();
  return data;
}, { secUid: userSecUid });
```

| Criterion | Rating |
|-----------|--------|
| **Difficulty** | 2/5 (moderate -- need to extract secUid) |
| **Reliability** | 4/5 (uses TikTok's own signing infra) |
| **Speed improvement** | ~60s -> ~10-15s (no scrolling, no tab clicking) |
| **Risk** | Low -- same mechanism TikTok's own frontend uses |

**Why it works:** When `fetch()` is called from within `page.evaluate()`, it executes in the browser's JavaScript context where TikTok has already:
1. Loaded its `byted_acrawler` library
2. Patched `window._fetch` / `window.fetch` with signature interceptors
3. Set up msToken and other session tokens in cookies

The browser's own fetch pipeline handles all signing automatically. This is exactly what davidteather's TikTok-Api does (via `run_fetch_script` / `generate_js_fetch`).

**Key requirement:** Must extract the user's `secUid` first. Options:
1. Parse it from the profile page HTML/JSON (embedded in `__UNIVERSAL_DATA_FOR_REHYDRATION__` script tag)
2. Extract it from the initial page API response during navigation

**Pagination:** Call in a loop with updated `cursor` values until `hasMore === false` or we find existing content.

**Verdict: USE THIS APPROACH.**

### C: TikTok Mobile API

**How it would work:** Call TikTok's mobile API endpoints (like Instagram's `i.instagram.com`) using mobile app User-Agent and headers.

| Criterion | Rating |
|-----------|--------|
| **Difficulty** | 5/5 (requires X-Gorgon, X-Argus, X-Khronos signatures) |
| **Reliability** | 2/5 (signatures change frequently) |
| **Speed improvement** | ~60s -> ~5s (no browser needed) |
| **Risk** | Very high -- mobile signatures even harder than web |

**Why it fails:** TikTok's mobile API uses **different** cryptographic signatures (X-Gorgon, X-Argus, X-Khronos) that are generated by the native app. These use Protocol Buffers, SM3 hashing, and Simon cipher encryption. They require valid `device_id`, `install_id`, and Android device parameters. Reverse-engineering and maintaining these signatures is a full-time job.

**Verdict: DO NOT USE.** Far more complex than the web API, and no existing Node.js library maintains these signatures reliably.

### D: Node.js Signature Generation

**How it would work:** Reimplement X-Bogus generation in pure Node.js, then make direct HTTP requests without a browser.

| Criterion | Rating |
|-----------|--------|
| **Difficulty** | 4/5 (algorithm is documented but obfuscated) |
| **Reliability** | 2/5 (TikTok updates algorithm periodically) |
| **Speed improvement** | ~60s -> ~2s (no browser at all) |
| **Risk** | Very high maintenance burden |

**Why it's not recommended:** While the X-Bogus algorithm has been reverse-engineered ([int4444/tiktok-api](https://github.com/int4444/tiktok-api)), TikTok regularly updates it. Even the most popular library (davidteather's TikTok-Api with 5k+ stars) abandoned native signing in favor of browser-based signing because maintaining native implementations against TikTok's obfuscation updates was unsustainable.

**Verdict: DO NOT USE.** The maintenance burden is not worth the 10s speedup over approach B.

### E: Hybrid Interception (current approach, optimized)

**How it would work:** Keep the `page.on('response')` interception but navigate directly to a URL that triggers the `favorite/item_list` call, skipping tab clicking and scrolling.

| Criterion | Rating |
|-----------|--------|
| **Difficulty** | 2/5 (straightforward changes) |
| **Reliability** | 3/5 (depends on TikTok triggering the API on page load) |
| **Speed improvement** | ~60s -> ~20-30s (still needs page loads) |
| **Risk** | Medium -- may not trigger API call without user interaction |

**This is a fallback** if approach B doesn't work because TikTok's fetch interceptor blocks programmatic calls.

**Verdict: FALLBACK OPTION** if B fails during implementation.

## Recommended Approach

**Use Approach B: `page.evaluate` + `fetch` from browser context.**

### Why This Is the Right Choice

1. **Same mechanism as the proven TikTok-Api library** (v7.2.1, actively maintained, released October 2025). It uses `page.evaluate()` to call `fetch()` within TikTok's signed browser context.

2. **Eliminates all fragile DOM interaction:** No tab clicking (5 methods!), no scrolling (15 iterations), no screenshot debugging. Just: navigate, wait for JS, call API, parse JSON.

3. **4-6x speed improvement:** From ~60s to ~10-15s per user. The browser still launches (unavoidable), but the actual data fetching is a single API call with pagination, not scroll-based interception.

4. **Minimal code change:** The `syncUserTikTok` function changes from ~300 lines of DOM automation to ~100 lines of API calls.

5. **Self-maintaining signatures:** TikTok's own JavaScript generates the signatures. When TikTok updates its signing algorithm, our code doesn't need to change -- the browser loads the updated JS automatically.

### Why Not Go Full Instagram-Style

The Instagram approach (`context.request.get()`) works because Instagram's mobile API trusts simple cookie + header authentication. TikTok does not. TikTok requires signatures generated by its client-side JavaScript VM. This is a fundamental architectural difference, and no amount of header manipulation will bypass it.

The browser is the price of admission for TikTok. Accept it and optimize within that constraint.

## Implementation Notes

### Key Code Changes

**Step 1: Extract secUid from profile page**

After navigating to `https://www.tiktok.com/profile`:
```typescript
const secUid = await page.evaluate(() => {
  // TikTok embeds user data in a script tag
  const scriptEl = document.getElementById('__UNIVERSAL_DATA_FOR_REHYDRATION__');
  if (scriptEl) {
    const data = JSON.parse(scriptEl.textContent || '{}');
    // Path may vary; inspect actual page structure
    return data?.__DEFAULT_SCOPE__?.['webapp.user-detail']?.userInfo?.user?.secUid;
  }
  return null;
});
```

**Step 2: Call favorite/item_list via page.evaluate**

```typescript
const allVideos: TikTokVideo[] = [];
let cursor = 0;
let hasMore = true;

while (hasMore && allVideos.length < maxVideos) {
  const result = await page.evaluate(async (params) => {
    const url = `https://www.tiktok.com/api/favorite/item_list?secUid=${params.secUid}&count=35&cursor=${params.cursor}`;
    try {
      const resp = await fetch(url);
      return await resp.json();
    } catch (e) {
      return { error: String(e), itemList: [], hasMore: false };
    }
  }, { secUid, cursor });

  if (result.itemList) {
    allVideos.push(...result.itemList);
  }
  hasMore = result.hasMore;
  cursor = result.cursor || 0;
}
```

**Step 3: Remove all DOM interaction code**

Delete:
- Cookie popup dismissal
- Liked tab clicking (all 5 methods)
- Scrolling loop
- Response interception (`page.on('response')`)
- Debug screenshots
- Tab detection JavaScript

**Step 4: Keep browser launch and cookie injection**

The browser context is still required for:
- Cookie injection (session authentication)
- Loading TikTok's JS (signature generation)
- Executing `page.evaluate()` in TikTok's signed context

### Extracting secUid

The `secUid` is critical -- it identifies which user's likes to fetch. Options:

1. **From `__UNIVERSAL_DATA_FOR_REHYDRATION__`**: Most reliable. TikTok embeds full user data in this script tag on the profile page.

2. **From URL after redirect**: When navigating to `/profile`, TikTok redirects to `/@username`. The secUid can be extracted from the page's embedded data at that point.

3. **Store it during cookie capture**: When the user connects TikTok in the iOS app, extract and store their secUid alongside cookies. This avoids needing to parse it on every sync.

**Recommendation:** Use option 1 for initial implementation, then add option 3 as an optimization to save one parse step.

### Fallback Strategy

If `page.evaluate(fetch(...))` is blocked by TikTok (e.g., they detect programmatic fetch calls), fall back to:

1. **Sign the URL explicitly:** Call `window.byted_acrawler.frontierSign(url)` via `page.evaluate()` to get the X-Bogus, then append it to the URL and make the request.

2. **Navigate to the API URL directly:** Use `page.goto('https://www.tiktok.com/api/favorite/item_list?...')` and parse the response from `page.content()`.

3. **Keep scroll-based approach as last resort:** The current code already works; it's just slow.

## Risks and Mitigations

### Risk 1: TikTok's patched fetch rejects programmatic calls
**Likelihood:** LOW (20%)
**Impact:** Implementation blocked
**Mitigation:** Fall back to explicit signing via `byted_acrawler.frontierSign()`, or navigate directly to API URL
**Detection:** Test immediately during implementation

### Risk 2: secUid extraction fails (TikTok changes page structure)
**Likelihood:** MEDIUM (30%)
**Impact:** Cannot identify user for API call
**Mitigation:** Store secUid during initial cookie capture in iOS app; or extract from multiple possible locations (script tag, API response, URL redirect)
**Detection:** Monitor for `secUid === null` in sync logs

### Risk 3: TikTok adds CORS restrictions to API from same-origin fetch
**Likelihood:** VERY LOW (5%)
**Impact:** Browser fetch fails
**Mitigation:** The API is same-origin (`www.tiktok.com`), so CORS doesn't apply. Only cross-origin requests face CORS. This risk is negligible.

### Risk 4: Likes must be set to public
**Likelihood:** EXISTS (user-dependent)
**Impact:** API returns empty results
**Mitigation:** This is the SAME limitation as the current scroll-based approach. Not a regression. Detect and notify users.
**Detection:** `itemList.length === 0` when user has known liked content

### Risk 5: Rate limiting on rapid API calls
**Likelihood:** LOW (15%)
**Impact:** Temporary blocks
**Mitigation:** Add delays between pagination requests (500ms-1s). Current approach already takes 60s, so even with delays, this is faster.
